# ChankoBlog Web Design Guidelines

这份文档用于每次进行网页、后台管理界面、组件或视觉原型设计前的约束检查。项目定位为带后端管理的 Nuxt 4 博客，因此默认设计目标是：内容清晰、管理高效、品牌一致、克制但不简陋。

## 1. 设计前置流程

在写任何页面代码前，先用自然语言声明本次采用的设计系统，并等待确认或根据上下文形成明确记录。

必须先给出以下 `Design Decisions`，不得跳过或只写“沿用项目风格”：

```markdown
Design Decisions:
- Color palette: [primary / secondary / neutral / accent]
- Typography: [heading font / body font / code font]
- Spacing system: [base unit and multiples]
- Border-radius strategy: [large / small / sharp]
- Shadow hierarchy: [elevation 1-5]
- Motion style: [easing curves / duration / trigger]
```

填写要求：

- `Color palette` 必须说明主色、辅助色、中性色、强调色各自承担的语义，不能只列色值。
- `Typography` 必须说明标题字体、正文字体、代码或数字字体，以及它们在内容页和后台界面中的使用边界。
- `Spacing system` 必须声明基础单位和可用倍数，默认使用 8pt grid；如偏离 8pt，必须说明原因。
- `Border-radius strategy` 必须说明整体圆角倾向：大圆角、小圆角或锐利边缘，并定义按钮、输入框、卡片、弹窗的层级。
- `Shadow hierarchy` 必须定义 1-5 级阴影或明确说明哪些层级不使用阴影；后台界面优先用边框和背景层次。
- `Motion style` 必须说明缓动曲线、持续时间和触发方式；若页面不需要动效，也要声明“仅保留状态过渡”。

扩展说明：

- 配色：品牌色、辅助色、强调色、中性色，以及是否使用 `oklch()` 派生。
- 字体：标题、正文、代码或数字字体的选择依据。
- 间距：采用 8pt grid，优先使用 `8, 16, 24, 32, 48, 64, 96`。
- 圆角：按钮、输入框、卡片、弹窗的圆角层级。
- 边框与阴影：后台界面优先用边框和层级，不滥用重阴影。
- 图标：使用项目已有图标库；没有图标库时先确认，不用 emoji 代替。
- 数据策略：真实数据、接口数据、明确空状态或待接入状态，不编造假数据。
- 实现目标：Nuxt 页面、Vue SFC、后台管理组件、原型页面或其他。

示例记录：

```markdown
Design Decisions:
- Color palette: primary 使用品牌主色承载链接、主按钮和当前状态；secondary 用于次级导航和辅助图形；neutral 使用统一灰阶承载背景、文字和边框；accent 只用于警示、选中或关键提示。
- Typography: heading font 用于页面标题和文章标题；body font 用于正文、表单和列表；code font 只用于代码块、技术标识和等宽数据。
- Spacing system: 8px 为基础单位，常用 1x/2x/3x/4x/6x/8x/12x，不使用零散间距。
- Border-radius strategy: small，按钮和输入框 6-8px，卡片不超过 8px，弹窗可略高但保持克制。
- Shadow hierarchy: elevation 1-2 用于悬浮和弹层，普通卡片不用重阴影，elevation 3-5 只保留给 modal、drawer、popover。
- Motion style: 160-240ms 状态过渡，使用 ease-out 或 cubic-bezier，触发来源限于 hover、focus、展开收起和进入离开。
```

设计节奏：

1. 先描述设计系统。
2. 尽早给出 v0 半成品，用于确认方向。
3. v0 只保留必要结构，不用无意义模块填满空间。
4. 方向确认后再补完整状态、响应式、交互和数据接入。

## 2. 色彩系统

颜色必须有来源，禁止凭空造色。

优先级：

1. 项目已有品牌色、主题变量、设计 token。
2. 由品牌色通过 `oklch()` 派生的浅色、深色、弱化色、边框色和背景色。
3. 必要的中性色，同样应在统一灰阶系统中定义。

限制：

- 避免紫色、粉色、蓝色渐变背景，除非品牌规范明确要求。
- 避免深蓝黑底加霓虹高光的赛博风格。
- 避免 5 种以上无结构色彩同时出现。
- 每个页面应控制在：1 个主色、1 个辅助色、1 个强调色、灰阶系统。
- 图表颜色必须表达语义，而不是装饰。

建议写法：

```css
:root {
  --brand-h: 120;
  --brand-primary: oklch(0.55 0.18 var(--brand-h));
  --brand-primary-soft: oklch(0.92 0.05 var(--brand-h));
  --brand-primary-strong: oklch(0.38 0.16 var(--brand-h));

  --gray-50: oklch(0.98 0.002 var(--brand-h));
  --gray-100: oklch(0.95 0.004 var(--brand-h));
  --gray-300: oklch(0.86 0.006 var(--brand-h));
  --gray-700: oklch(0.38 0.01 var(--brand-h));
  --gray-900: oklch(0.18 0.012 var(--brand-h));
}
```

具体色相必须来自真实品牌色，不使用上方示例的数值作为默认方案。

## 3. 字体与排版

优先遵守项目或品牌已有字体。没有明确规范时，再根据博客与后台管理的气质选择。

原则：

- 最多使用 2 个字体家族。
- 标题、正文、辅助信息要有明确层级。
- 标题与正文大小差距要足够明显，避免信息主次不清。
- 正文从 16px 到 18px 起步，后台表格和辅助信息可更紧凑，但不能影响可读性。
- 避免所有文字尺寸接近，导致用户无法快速定位重点。
- 避免用过度常见的 AI demo 字体组合制造廉价感；如已有品牌规范，则品牌优先。

排版检查：

- 页面主标题是否是视觉锚点。
- 正文行高是否舒适。
- 表格、表单、卡片标题是否比正文更清晰。
- 长文本是否使用 `text-wrap: pretty` 或合理宽度控制。

## 4. 间距、布局与留白

统一使用 8pt spacing system。

常用间距：

- `8px`：紧密元素间距、图标与文字间距。
- `16px`：表单项、工具栏、卡片内部基础间距。
- `24px`：组件组之间、列表区域间距。
- `32px`：页面区块间距、主要容器内边距。
- `48px`：大区块分隔。
- `64px` / `96px`：营销页、文章页或大视觉区域。

原则：

- 不用随意数值拼凑布局。
- 留白不是空洞，留白用于建立层级和阅读节奏。
- 不通过增加虚拟模块、占位文案或假数据来填满页面。
- 后台管理界面可以信息密度更高，但仍需清楚的分组、对齐和扫描路径。
- 对齐方式以左对齐为主；居中只用于 hero、空状态或特殊强调场景。

## 5. 组件与视觉规则

卡片：

- 避免左侧彩色色条卡片。
- 卡片用于重复内容、弹窗、工具区或明确的内容容器。
- 不要在卡片里再套卡片。
- 圆角保持克制，默认不超过 8px，除非项目系统另有规定。

图标：

- 不使用 emoji 当图标。
- 优先使用项目已有图标库。
- 图标按钮要有明确语义和 hover/focus 状态。
- 缺少图标资源时，应标记为待接入或先确认图标来源，不临时硬画复杂 SVG。

图像与 SVG：

- 避免复杂硬编码 SVG，尤其是人物、场景、产品图。
- 品牌、产品、头像、封面等应使用真实资源或明确的上传/空状态。
- 不用 CSS 轮廓或手绘 SVG 冒充真实产品资产。

按钮与表单：

- 按钮用于明确命令，图标按钮用于工具操作。
- 表单状态必须覆盖默认、hover、focus、disabled、loading、error。
- 后台管理中的表格、筛选、分页、抽屉、弹窗应优先遵循现有组件体系。

## 6. 内容与数据规则

禁止堆砌假数据。

允许的内容来源：

- 真实接口数据。
- 项目已有 mock 规范。
- 用户明确提供的示例内容。
- 明确标注的空状态、加载状态、错误状态、待接入状态。

不允许：

- 虚拟统计数字。
- 虚构用户评价。
- 假 logo 墙。
- 无功能意义的模块。
- 为填充空间添加的占位符文本。

每个页面元素都必须回答：

- 它解决什么问题？
- 它服务哪个用户动作？
- 删除它后页面是否变差？

如果答案不清楚，应删除或延后。

## 7. 需要避免的常见问题

每次设计自检时，逐项检查：

1. AI 科技陈词滥调：避免渐变球、数字雨、蓝色电路板、机器人脸。
2. 字号层级不足：标题和正文过于接近会削弱扫描效率。
3. 颜色过多：控制主色、辅色、强调色和灰阶结构。
4. 间距不一致：统一使用 8pt grid。
5. 留白不足：不要把每个区域都塞满。
6. 字体过多：最多 2 个字体家族。
7. 对齐混乱：默认左对齐，少量场景可居中。
8. 装饰压过内容：背景、纹理、阴影必须通过删除测试。
9. 赛博霓虹过度：避免默认深蓝黑加荧光边的开发工具套路。
10. 信息密度不匹配：后台、文章页、封面、图表各自采用合适密度。

删除测试：

> 如果移除某个装饰、模块或动效后，设计没有变差，就移除它。

## 8. 可借鉴的实现模式

以下模式可作为实现参考，但必须服从本项目设计系统，不能直接照搬示例颜色、字体或假数据。

- 响应式演示页：适合固定比例页面、汇报页或设计稿展示。
- 设备模拟框：适合移动端或浏览器界面预览。
- Tweaks 调参面板：适合多方案对比、主题调整和设计探索。
- 动画时间轴：适合复杂演示动效或可播放原型。
- Design Canvas：适合并排展示多个视觉方案。
- 暗色模式：必须基于品牌色和中性色派生，不做默认赛博霓虹。
- 数据可视化：优先 Chart.js 或项目已有图表库，颜色必须有语义。

## 9. Nuxt 4 项目实现约定

实施前先检查：

- `package.json` 中的框架、UI 库、脚本和依赖。
- `nuxt.config.ts` 的配置。
- `app/` 下的入口、布局、页面、组件和全局样式。
- 是否已有主题变量、设计 token、图标库、组件库。

实现原则：

- 使用 Vue SFC 和 Nuxt 约定目录。
- 设计 token 放在统一的全局样式或主题文件中，不把颜色和间距散落在各组件里。
- 后台管理页面优先考虑表格、筛选、分页、批量操作、抽屉、弹窗、空状态和错误状态。
- 用户端博客页面优先考虑阅读体验、文章层级、目录、标签、作者信息、响应式排版。
- 不在没有确认的情况下新增大段无关页面或模块。

## 10. 交付前检查清单

交付前必须确认：

- 已声明设计系统。
- 已使用品牌色或从品牌色 `oklch()` 派生。
- 没有凭空新增颜色。
- 没有紫粉蓝渐变背景。
- 没有 emoji 图标。
- 没有左侧彩色色条卡片。
- 没有复杂硬画 SVG。
- 没有虚构数据、虚拟评价或假统计。
- 没有为了填空间而添加的占位文本或无意义模块。
- 字号层级清晰。
- 间距来自统一尺度。
- 对齐方式一致。
- 装饰没有压过内容。
- 关键交互有 hover、focus、active、disabled、loading、empty、error 等必要状态。
- 移动端和桌面端没有文字溢出或元素重叠。
- 后台管理界面保持可扫描、可操作、可维护。
